So, what is Git?
Git is a type of version control system (VCS) that makes it easier to track changes to files. For example, when you edit a file, git can help you determine exactly what changed, who changed it, and why.

What makes git unique?
  • Git branches are cheap and easy to merge. This facilitates the feature branch workflow popular with many Git users.
  • Git is a distributed version control system. Every user gets their own local repository, complete with a full history of commits.
  • Having a full local history makes Git fast, since it means you don’t need a network connection to create commits, inspect previous versions of a file, or perform diffs between commits.
The Git Places Model

Where does git store your files? Contrary to what you'd think, it's not just your computer and the cloud. In Git, there are five places your source can exist:
  • A Stash
  • A Local working directory.
  • An Index (or staging area)
  • a local repository
  • A remote repository
The first four are located in each user's personal computer, whereas the remote repository is shared by all users. Scroll down to learn more about them!
Workspace
This is the local working directory for your local repository whose contents are checked out in the branch you are working in.
Index
These include the files you want to commit. This is considered the staging area where the files you intend to commit are temporarily tracked. Before you “commit” (checkin) files, you need to first add them to the index. Also called "current directory cache", "staging area", "cache" or "staged files".
Local Repository
Git local repository is the one on which we will make local changes, typically this local repository is on our computer. It contains a subdirectory named `.git` that contains all of your necessary repository files — a Git repository skeleton. The default name for a local branch is "master".
Remote Repository
Git remote repository is a remotely hosted service which stores a version(s) of your code, ensuring all your changes are available for other developers. The default name for a remote is "origin".
Stash
This is a place where you can store your code changes while you are trying to pull in the latest version from the remote repository.
Commands Move Files
While collaborating with people on the same repository, we regularly encounter situations when we need to either update our local folders to take up changes that someone else has made. Similarly, we push our code to the remote repository when we want to update the codebase so others can access the new version. It is therefore, critical to understand the idea of code flowing upstream (from your computer to the cloud repository) and downstream (from the cloud repository to your computer). Scenario 1: Share with the world!

Let's say you are working on a blog post. You create, edit and save a new file . How do you let the world know about it?

Try git add to start tracking files, which you intend to commit and share. Make as many changes, as you wish and keep adding it to the staging area.

Once you are ready, use git commit to move these changes out of the temporary index to your local repository. Each commit creates a new commit node in a commit tree structure stored in the .git subdirectory. If you wish to "undo" a commit, Git can track these changes in the tree.

After committing multiple versions of the file, use git push to send the file over to the cloud repository. Your blog post is now available to all your collaborators!

Scenario 2: Bringing New Opinions

Let's say that you finalized your version of the blog post and pushed it to the cloud. Your team can now makes changes, and your co-author pushes a new version to the cloud. How do you see the combined file? With downstream commands!

First, you can carefully bring the changes to your local repository using git fetch . This doesn't change your workspace, so you can potentially still see and/or change your old blog post. Then, with git merge the combined file is now reflected in your directory and a node is automatically added in the commit tree.

Alternatively, if you feel lazy, you can use git pull that does a fetch and merge combined.The caveat: Merge and pull work on a clean working directory, i.e. without any uncommitted/ untracked changes.

Scenario 3: Edit, Interrupted

Often, the cloud repository changes while you are working on files as well. Let's create a simple website for a team project. You create and start editing an HTML file, as your co-author pushes an image to the repository. How do you bring this image file into your current directory?

If you try to git pull like earlier, it won't work! That's because your working directory has untracked changes!

You don't want to delete this HTML file, or commit/push it yet. With git stash , you can store away all these uncommitted and untracked changes temporarily. Try the pull command again to see the difference.

Restoring your work in progress, is now easy. Use git pop to clear out the stash, and bring back your changes to the working directory.
